# Assumptions Audit Log

## Fixed Assumptions

### 1. Committee Code Map was Static (Fixed: dynamic from API)
- **What it assumed:** The 25 committee codes (H01-H24, S01-S13) never change
- **How it broke:** New committees could be created, existing ones renamed or merged between sessions
- **Fix:** Rebuilt from Committee API at runtime with static fallback + drift alerting

### 2. LOCAL_LEXICON was Hardcoded (Fixed: derived from API)
- **What it assumed:** Committee name aliases are stable across sessions
- **How it broke:** Committee names change (e.g., "Courts of Justice" was H02 in old map, is actually H08)
- **Fix:** Auto-generated from Committee API names, splitting on commas/ands

### 3. Parent Fallback Used startswith Heuristic (Fixed: ParentCommitteeID)
- **What it assumed:** Subcommittee names always start with parent committee name
- **How it broke:** Could match unrelated committees with similar name prefixes
- **Fix:** Validate against ParentCommitteeID from Committee API response

### 34. Schedule sub-panels not recognized as children (HCJ-Civil, HCJ-Criminal, etc.)
- **What broke:** House Courts of Justice, House Appropriations, House Finance, and House Labor and Commerce have Schedule API entries with hyphen-suffixed names ("House Courts of Justice-Civil", "House Appropriations - Health and Human Resources Subcommittee") that carry concrete times. The parent entries have "Time TBA". ### 35. api_schedule_map overwrites concrete times with TBA on duplicate entries
- **What broke:** Multiple Schedule API entries can exist for the same date+committee. `api_schedule_map[map_key] = {...}` overwrites unconditionally, so a later entry with "Time TBA" kills an earlier entry with a concrete time.
- **How it was caught:** Investigation of how duplicate entries are ordered in the API response.

### 36. Subcommittee vote refids (H14003V...) not parsed by resolve_committee_from_refid
- **What broke:** The vote regex `r'^([HS])(\d{1,2})V\d+'` only matched parent vote refids like `H14V2610034`. Subcommittee vote refids like `H14003V2610048` (parent code H14 + subcommittee suffix 003 + V + vote ID) have 3 extra digits before the V, which the regex didn't allow. Result: 1,637 subcommittee refids returned `(None, None)`. Without a refid match, actions fell to lexicon (which also fails for "H Subcommittee recommends..." since no committee name appears in the text) → Memory Anchor → often wrong committee → Journal Entry → Ledger.
- **How it was caught:** Post-PR#16 X-Ray showed 453 Ledger meeting bugs, all COMMITTEE_DRIFT tagged with subcommittee actions. Sample refids like `H24001V2612630` confirmed the pattern.
- **Data:** 1,637 subcommittee refids missed across 14 House committees. Top: H02 (245), H14 (203), H08 (179), H24 (159).
- **Fix:** Expanded regex to `r'^([HS])(\d{1,2}?)(?:\d{3})?V\d+'` — non-greedy `\d{1,2}?` captures the parent code (1-2 digits), optional `(?:\d{3})?` matches the strictly 3-digit subcommittee suffix. Non-greedy is critical: greedy `\d{1,2}` would consume a digit from the subcommittee suffix on 1-digit parent codes (e.g., S2001V → S20 instead of S2). Verified: zero false positives against all non-committee refid formats.
- **Lesson:** When an ID format has structural components (parent + subcommittee + vote), the regex must account for ALL components, not just the most common pattern. Test against the full range of real data, not just a few examples. When consecutive numeric fields have variable width, greedy quantifiers are dangerous — always test with the minimum-length variant of each field.

### 37. "Prefiled and ordered printed; Offered..." misclassified as meeting action
- **What broke:** X-Ray `MEETING_ACTION_PATTERNS` contains the substring "offered" to catch legitimate committee/floor amendment offerings ("H House committee offered"). Query of HISTORY.CSV showed 3,682 total "offered" rows in session 261: 2,042 "Prefiled and ordered printed" (admin), 698 "H House subcommittee offered" (meeting), 526 "S Senate committee offered" (meeting), 184 "H House committee offered" (meeting), rest misc. The prefiled chunk is the dominant noise source.
- **Fix:** Added `"prefiled and ordered printed"` to `ADMIN_OVERRIDE_PATTERNS` in both `pages/ray2.py` and `calendar_xray.py`. Override is checked before `MEETING_ACTION_PATTERNS`, so prefiled rows now classify as administrative. Legitimate committee/subcommittee amendment offerings are unaffected (no "prefiled" substring).
- **Scope note:** This is a classification-hygiene fix. It corrects the meeting/admin denominator in X-Ray but does not by itself fix the ~41 genuine committee-amendment offerings that are in Ledger Updates due to missing committee convene times. That investigation continues.
- **Lesson:** Substring-based classification must be audited against real phrase frequency, not just the token meaning. "offered" as a bare word is a meeting verb; "prefiled and ordered printed" is a clerk phrase that happens to contain it. The specificity-wins override pattern (ADMIN_OVERRIDE_PATTERNS) is the right place for these exceptions.

### 42. PR-C3 LegislationEvent fallback: N+1 fetch (no per-cycle response cache)
- **What broke:** PR #30 merged 2026-04-25 ([[log#2026-04-25-pr--pr-c3-round-2--pr-30-merged-and-reverted-same-day]]). On the very next 15-min cycle the GitHub Actions worker hung 11+ min vs normal ~2 min and was manually canceled. Reverted on main as commit `246cba5`.

Telemetry from inside the cycle was unavailable — the cycle never finished writing its metrics row, so X-Ray showed nothing.
- **Root cause:** `_resolve_via_legislation_event_api` cached the `LegislationID` lookup (the bill-number → ID hop) per (bill, session) but did NOT cache the second HTTP call — `GetPublicLegislationEventHistoryListAsync`. The endpoint returns the bill's whole event history in one shot, so a *single* fetch covers every action_date for that bill. Yet every `journal_default` row in HISTORY.CSV that hit the gate triggered a fresh fetch. With ~3,000 unique bills and ~10,000+ journal_default rows across the full session window, that's thousands of redundant HTTP calls. Combined with `urllib3.Retry(total=4, backoff_factor=2)` on 429s, LIS WAF rate-limiting cascaded into 40s+ stalls per affected request — the cycle's wall-clock blew past the 15-min Actions schedule.
- **Audit hole that let this through:** Pre-merge testing exercised matcher correctness (4 Class-1 bug cases + HB1 multi-event + abstain safety nets, 11/11 passing) but did NOT exercise the gate's selectivity at session scale or the per-cycle HTTP-call count. The reviewer playbook for new fallback paths needs a **candidate-set sizing check** — for any new conditional that fires HTTP, count `n_eligible_rows` × `expected_calls_per_row` against the session window before merge. The matcher was correct in isolation; correctness in isolation is not correctness in production.
- **Fix:** PR-C3.1 (PR #31) adds `_legislation_event_cache` per-cycle, mirroring `_legislation_id_cache`. Keys are `(bill_num, session_code_5d)`. Successful fetch stores the events list; any failure path (network exception, non-200, parse error, non-dict shape) stores `[]` to negative-cache and suppress retry storms within the cycle. Categorized `push_alert` with `dedup_key` still fires on miss so the failure remains visible. Two new regression tests prove cache reuse and negative-cache (`test_pr_c31_event_cache_prevents_refetch`, `test_pr_c31_negative_cache_suppresses_retry_on_failure`).
- **Lesson:** When a fallback adds a new HTTP call inside a row-by-row loop, it is NOT a fallback — it is a multiplicative cost. The unit of caching must match the unit the source returns. Here the LIS endpoint's response unit is "the bill's whole history" but the worker's call unit was "each row." Whenever those don't match, the response MUST be cached at the source's natural granularity, not the consumer's loop granularity. Add this to the pre-push audit when any new fallback fires HTTP from inside a loop.

### 44. Revert-of-merge resolved via merge with `-s ours` (not `-X ours`, not `git revert <revert>`)
- **What broke:** PR-C3.1 was branched from the PR-C3 tip (`f5745c4`) to preserve a single review surface. After main reverted PR #30 (commit `246cba5`), GitHub showed PR #31 as conflicted. Two attempts to clear the conflict each had a non-obvious failure mode before the third worked.

- **Root cause:** This is the canonical "revert-of-faulty-merge" pattern documented in `git-scm.com/docs/howto/revert-a-faulty-merge`. Merge-base of branch and main was `f5745c4` (because main once contained PR-C3 via merge `4d398ac`). From base, main's diff is "delete PR-C3 lines" and branch's diff is "keep them, with PR-C3.1 enhancements". Both sides modified the same lines from the base, in opposite directions — true conflict.
- **Failed attempt 1: `git revert 246cba5` on the branch.** Conflicts resolved by taking HEAD ("ours") and the result was a zero-diff commit — `git status` reported "nothing added to commit". The branch HEAD already contained everything `246cba5` removed (plus the PR-C3.1 enhancements), so the revert tried to re-add content that was already there. An empty commit on the branch tip would not have changed file content on either side, leaving GitHub's three-way merge state — and conflict — unchanged. Aborted.
- **Failed attempt 2: `git merge -X ours --no-ff origin/main`.** The `-X ours` *strategy-option* only resolves textual conflicts in our favor. For non-conflict regions where "ours" hadn't diverged from base but "theirs" had (the revert deleted lines, ours had no diff vs base for those lines), git auto-applied "theirs" change as a clean non-conflict — silently un-applying module-level constants like `LEGISLATION_EVENT_HEADERS` that the resolver depended on. Caught by tests dropping from 13/13 to 5/13 with `NameError: name 'LEGISLATION_EVENT_HEADERS' is not defined` traced via `sys.settrace`. Aborted.
- **Working fix: `git merge -s ours --no-ff origin/main`** (commit `a2bb618`). The `-s ours` *strategy* (not strategy-option) discards "theirs" tree entirely while still recording main as a merge parent. Branch's tree stayed byte-identical to pre-merge (13/13 still passing); merge-base of branch and main shifted to `246cba5`; subsequent `branch → main` merge has zero divergence; no force-push, no history rewrite.
- **Lesson:** When a feature branch built on the pre-revert state needs to merge into a main that has reverted those changes, the right operation is `git merge -s ours` on the branch — NOT `git revert <the-revert>` (zero-diff no-op when ours already has the content), and NOT `git merge -X ours` (silent corruption on non-conflict deletions). The strategy form is for "I want my tree, but I want main in my history so the conflict goes away." The strategy-*option* form is for "merge normally, but tiebreak conflicts toward ours" — different semantics, dangerous default for revert-of-merge. **Process check:** after any merge with non-trivial conflict resolution, RUN THE TESTS before pushing. The `-X ours` corruption was caught only because tests fired before the push button was reached. If we'd pushed and let CI run, we'd have polluted PR #31's history with a broken commit and probably triggered another bot review cycle.

### 47. Queue-with-cap requires explicit overflow handling — items beyond the cap don't disappear, they continue through the system (PR-C7 review fix)
- **What broke:** PR-C7's first commit (`70f14f8`) introduced a refresh queue capped at `LEGEVENT_FETCHES_PER_CYCLE = 500` for hydrating the cross-cycle LegEvent cache. The queue building was correct (Tier A → B → C, cap enforced, overflow telemetry in place). But the row-loop integration assumed *implicitly* that "any bill that needs fetching is in the cache after hydration." That assumption broke for **Tier A overflow bills** — bills that needed hydration but exceeded the 500 cap (queued for next cycle). When a row from one of those bills reached the journal_default LegEvent fallback at `calendar_worker.py:3110`, the resolver's existing PR-C3.1 cache check `if cache_key in legislation_event_cache:` was False (no entry), and the resolver fell into its network-fetch path. **The 500-fetch cap was bypassed.** Worst case: cold-start cycle 1 with 1,500 Tier A overflow bills could fire 1,500 ad-hoc fetches from the row loop, recreating the [[#42|PR-C3 hang]] vector this entire PR was designed to eliminate.
- **How it was caught:** **Codex P1** and **Gemini critical** flagged the same issue independently in PR review on commit `70f14f8`. Both correctly identified that overflow bills bypass the cap. Two-bot agreement on the same flaw is a strong signal that the issue is real and not a false positive.
- **Fix:** PR-C7 review fix commit `45c72b5` adds a "negative-cache seed" loop AFTER the hydration phase, BEFORE the row iteration. For every candidate bill that did NOT make the queue (Tier A overflow, OR skipped on terminal/fresh), seed both:
    `_legislation_id_cache[(bill, session_5d)] = ""` — falsy value; resolver bails at `if not legislation_id` BEFORE any network fetch
    `_legislation_event_cache[(bill, session_5d)] = []` — empty list; resolver bails at `if not matching` (defense in depth)
  The resolver short-circuits without making any HTTP request. The row loop is now strictly cache-lookup-only regardless of which tier a bill is in, or whether it was hydrated this cycle. New telemetry counter `legevent_overflow_no_fetch` tracks how many bills were seeded with negative cache so the operator can see "this cycle suppressed N row-loop fetches that the prior PR-C7 design would have made."
- **Root cause (process-side):** I framed PR-C7 around the hydration phase ("the row loop is now lookup-only because hydration pre-populates everything") and didn't trace what happens to items that don't make the cap. The queue's cap enforcement was correct in the queue-building code, but the row-loop integration silently assumed hydration was complete. **The bug lived at the seam between two correctly-implemented components.**
- **Lesson:** **When introducing a queue with a per-cycle cap, items that exceed the cap do NOT disappear from the broader system — they continue through whatever logic consumes the queue's output. Explicit handling is mandatory.** Three patterns for handling overflow:
  1. **Negative cache (preferred when downstream consumer already has a "key absent" semantics):** seed the consumer's data structure with a value that signals "intentionally not fetched this cycle." Downstream calls short-circuit cleanly. (PR-C7's fix.)
  2. **Bypass branch:** add an explicit check at the consumer site that skips the consumer's expensive path if the item wasn't queued. Less elegant; couples consumer to queue.
  3. **Bounded-blocking:** make the consumer wait for the queue to drain. Defeats the point of the cap. Don't use.
  The first two are valid; pick based on which side has cleaner semantics. **Process upgrade:** when implementing a queue-with-cap pattern, the pre-push audit MUST trace one explicit overflow item end-to-end through the system. If the overflow item triggers any expensive operation downstream (network fetch, DB write, etc.), the cap is theoretical. Bot review caught this; the next instance might not.

### 50. Function-scope import shadows module-level binding for the entire function — PR-C7 cold-start brick (PR-C7.0.1 hotfix)
- **What broke:** PR-C7 merged at `c917d6de` 2026-05-05T00:01:55Z. The very first cycle on post-merge main (run 25350329090, `workflow_dispatch` at 2026-05-05T00:02:24Z, ~30s after merge) crashed with `UnboundLocalError: local variable 'timezone' referenced before assignment` at `calendar_worker.py:1893` inside `run_calendar_update()`. Worker bricked — every 15-min cycle would have failed identically until fixed. Last green cycle was 2026-05-04T23:35:53Z (run 25349432088 on pre-merge `30391238`).
- **Root cause:** PR-C7 added `from datetime import timezone` at `calendar_worker.py:2793` inside the LegEvent recovery block of `run_calendar_update()`. Python's scoping rule treats *any* name assignment (including `from X import Y`) as making the name local for the **entire enclosing function**, regardless of where in the function it appears. So the earlier references at lines 1893 and 1906 — which had been resolving to the module-level `from datetime import datetime, timedelta, timezone` at line 12 — suddenly became local references that hadn't been bound yet at the point of access. The module-level `timezone` was already in scope; the local re-import was redundant *and* poisonous.
- **How it was caught:** worker GitHub Actions run failure log immediately after merge. The cycle could not even reach the LegEvent block — it died at `_cycle_start_utc = datetime.now(timezone.utc)` at line 1893, very early in `run_calendar_update()`. Detection lag: ~30 seconds (next cycle after merge). Fix lag: ~5 minutes (PR #42 opened on commit `efe1a90`, one-line deletion).
- **How it escaped the 9-point pre-push audit:**
  1. Point 8 (Import Resolution from Subpage) checks `pages/*.py` import paths — function-scope import shadowing inside `calendar_worker.py` itself was not in scope.
  2. Parse-cleanliness was confirmed (`python -m py_compile`) but execution-cleanliness on the actual entry path was not. The bug ONLY surfaces at runtime when the function is invoked.
  3. The redundant import was syntactically harmless and visually plausible — a maintainer reviewing the diff sees `from datetime import timezone` next to `datetime.now(timezone.utc)` and assumes it's needed for that line.
- **Lesson:** **Never write a redundant `from X import Y` inside a function when `Y` is already imported at module level.** The redundancy isn't merely stylistic — it changes the scope of *every reference to `Y`* in the function, retroactively making the function local-bind. Even if today's references all fall after the local import, tomorrow's refactor that adds an earlier reference will break with `UnboundLocalError`. **Audit upgrade:** add Point 10 to the 9-point pre-push audit — *Function-Scope Shadow Check.* For any function containing a local `from X import Y` or `Y = ...` assignment, grep the function body for all references to `Y` and confirm every reference appears textually after the local binding. Better still: if `Y` is already at module level, just delete the local binding. **Process upgrade:** parse-clean is necessary but not sufficient. A 60-second pre-merge dry-run (worker invoked with `IS_DRY_RUN=true` env var, no sheet writes) would have caught this and any other cold-path runtime error. PR-C7's 558-line diff merited that step. Add to the workflow: post-CI, pre-merge, fire a `workflow_dispatch` dry-run on the head commit and confirm it reaches steady-state log lines before merging.

The specification was wrong because it was anchored to a transient value, not to a structural property.
- **How it was caught:** owner pattern-matched on the gap counter climbing past 24 hours and asked "stale data is unacceptable in a live tracking environment" — a directive that forced the calibration question to the surface. Reserve absolute thresholds for genuine catastrophes (the 500 floor here) where the floor IS structurally fixed (cardinality-of-bills × per-bill-row-cap × something). Treat any cycle-stable breaker trip as a CRITICAL calibration bug, not a transient. **Process upgrade:** breaker-carry-forward already surfaces the trip as a SYSTEM_ALERT one cycle later. Add a "trip-streak counter" — N consecutive trips without a successful cycle in between is a different failure mode than a single trip and should escalate (CRITICAL severity, distinct dedup_key) so monitors can distinguish "transient anomaly" from "stuck breaker." **Cross-reference:** this is the same shape of failure as #48 ("when a metric jumps suddenly at scale, sample-verify the metric's definition before scoping a fix") — both are "the metric's behavior changed, the static interpretation didn't follow." #48 was about diagnosis (we measured wrong); #53 is about action (we acted on a stale threshold). **Stale data is unacceptable in a live tracking environment** — the breaker exists to protect against bad writes, but a breaker that trips on healthy operation is itself the bad outcome.
- **Bot review fold-in (Codex P2, PR #45 commit `af4aa7e`):** the initial fix keyed delta-check activation on `last_known_good_meeting_unsourced > 0`, which silently conflated two semantically distinct states: (a) Y2 absent / unreadable / malformed → no trusted baseline, bypass delta-check (correct behavior), AND (b) Y2 = "0" → valid baseline of zero, delta-check should remain ACTIVE. Delta-check activation keys on the presence flag, not the numeric value. All failure modes (parse error, read error, empty cell) leave `present=False` → delta-check inactive + catastrophic floor still applies. **Lesson generalization:** **never encode "absence of a value" by a sentinel value of that same type.** "Y2 absent" and "Y2 = 0" must be distinguishable; using `0` as both the empty sentinel AND a legitimate value collapses information that the breaker needs. Decision rule: when a state cell can legitimately hold any value in its domain, presence must be tracked separately (a boolean flag, an `Optional[T]`, `not in dict` semantics, etc.) — NOT inferred from the value. **Audit upgrade #2:** add Point 15 to the pre-push audit — *Sentinel-Value Collision Check.* For any state cell read or persisted-value load with a default-on-failure path, ask: "is the default ever a legitimate runtime value?" If yes, track presence separately. **Same root class** as a category of bugs that bites Optional/Maybe types in any language — the explicit-presence pattern is the universal answer. **And:** treating bot review as a real signal paid off again — Codex caught a bug that wouldn't have surfaced until PR-C7.1 ships and the classifier drops to 0, by which point a regression-shaped outage would have been silently accepted and ratcheted into the baseline.

### 54. Minimizing the labor cost of a "small mapping table" is the failure mode of Standard #6 — bootstrapped SaaS cannot absorb routine maintenance even at low cadence (PR-C7.1 architectural pivot)

- **How it was caught:** owner pushback on the proposal. The pattern I was minimizing: a low-frequency interrupt is still an interrupt. SaaS economics requires the architecture to absorb routine variation deterministically; humans get pinged only for genuine anomalies that demand judgment.

The pattern decays into a maintenance treadmill at scale even when each individual addition is small.

### 56. Adding an optional column to a write-then-clear-trailing tab must APPEND, not insert mid-schema (PR-C7.0.6 Codex P2 fold-in, 2026-05-31)
- **What I did:** PR-C7.0.6 added an `EventCode` column to `LegEvent_Events`, inserting it at index 4 (between `EventDate` and `ChamberCode`) because that grouped the structural fields nicely.
- **Why it was wrong (Codex P2):** `_persist_legevent_cache` uses **write-then-clear-trailing** (PR-C7 Gemini review) precisely so a transient Sheets failure mid-persist leaves OLD data recoverable. But that means a failed cycle can leave the NEW 7-col header + first chunks alongside STALE old-6-col rows after the failed chunk. The reader is header-name-indexed, so under the new header it reads stale rows as `EventCode=<old ChamberCode>`, `ChamberCode=<old Description>`, `Description=""` — silent corruption of cached events until a refresh. Inserting the column mid-schema shifted every legacy column's effective index for stale rows.
- **Fix:** append `EventCode` LAST (after `Description`). Stale old-6-col rows then keep every legacy field at its original index; the absent trailing `EventCode` reads `""` via the existing bounds guard. Partial-write recoverability preserved.
- **Lesson:** **when evolving a schema that is persisted via a partial-write-tolerant pattern (write-then-clear-trailing, append-only logs, etc.), ADD new columns at the END, never insert among existing ones.** A name-indexed reader trusts the header's column positions; a partial write creates rows whose width predates the new header, and only end-appended columns degrade safely (absent → default) rather than misaligning every column after the insertion point. Same family as protobuf/Avro "only append fields, never renumber" and SQL "add nullable column at end." Directly relevant to PR-C7.1b (which may add a derived-category column to this same tab) and to 50-state scaling (more per-event fields). In-code rationale lives in the `LEGEVENT_EVENTS_HEADER` constant note.
